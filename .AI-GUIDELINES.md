# AI/KI Guidelines for LPO Documentation

**Purpose:** Enforce quality standards in markdown documentation and git commits.
**Applies to:** All AI/KI agents editing `.md` files or preparing commits in this repository.
**Last Updated:** June 8, 2026

## 1. Pre-Commit Validation Rules

All edits must pass pre-commit hooks before committing. Run this before any commit:

```bash
pre-commit run --all-files
```

### 1.1 Markdownlint Rules

The project uses `.markdownlint.yaml` with the following key rules:

| Rule | Code | Constraint | Example Violation |
| --- | --- | --- | --- |
| Line Length | MD013 | Max 119 characters per line | `This is a very long line that exceeds...` |
| Code Block Language | MD040 | All fenced code blocks must have language identifier | `` ``` `` (missing language) |
| Emphasis as Heading | MD036 | Don't use emphasis (`*text*`, `_text_`) where headings should be used | `*Total: ~25 minutes + Q&A*` |
| Duplicate Headings | MD024 | No duplicate heading text within same document | Two sections both named `### What matters most` |
| No Empty Links | MD042 | Links must have valid reference | `[text](#)` |

### 1.2 Other Pre-Commit Hooks

- **gitleaks:** Scans for secrets and credentials (API keys, tokens, passwords)
- **sphinx-build:** Validates reStructuredText syntax and cross-references
- **yaml-format:** Ensures YAML files are properly formatted

## 2. Markdown Linting Rules Applied

### 2.1 Line Length (MD013)

**Rule:** Keep regular prose lines at 119 characters or less.

**Configured exemptions in `.markdownlint.yaml` (MD013):**

- `code_blocks: false` (no line-length check inside fenced code blocks)
- `links: false` (no line-length check for links/URLs)
- `tables: false` (no line-length check for table rows)

**Readability best practice:**

- Even with exemptions, prefer narrower lines when practical.
- Keep prose and list text compact for easier review in terminals and diffs.
- Preserve long code/URLs/tables when wrapping would hurt clarity.

### 2.2 Code Block Language (MD040)

**Rule:** All fenced code blocks must have a language identifier

**When code blocks lack language:**

- Identify content type: `bash`, `yaml`, `python`, `json`, `markdown`, `text`, etc.
- Add immediately after opening backticks

### 2.3 Emphasis as Heading (MD036)

**Rule:** Use proper heading syntax (`##`, `###`) instead of emphasis for section titles

### 2.4 Duplicate Headings (MD024)

**Rule:** No duplicate heading text within the same document

**Resolver Pattern (recommended):** When a document logically needs repeated visible subheadings (for example, repeated
"✅ Advantages" under different parent sections), prefer keeping the visible heading text identical and make the source
unique using an invisible HTML comment or empty span. This satisfies MD024 while preserving readable headings.

Examples:

- `#### ✅ Advantages <!-- variant: A -->`
- `#### ✅ Advantages <!-- variant: B -->`

Alternatives:

- `#### ✅ Advantages <span id="advantages-a"></span>` — creates a hidden anchor and keeps visible text unchanged.
- Use visible qualifiers only when helpful for reading (e.g., "✅ Advantages (Variant A)").

### 2.5 No Empty Links (MD042)

**Rule:** Links must reference valid targets, not empty fragments

## 3. Prevention at Generation Time (Best Practice)

**Key Principle:** Avoid creating linting errors in the first place. Don't generate violating markdown and then fix it.

### 3.0 Before Starting

1. **Know the rules** – Review section 2 above (MD013, MD040, MD024, MD036, MD042)
2. **Check line length constraint** – 119 characters max in this project
3. **Plan abbreviations** for long titles/URLs before writing
4. **Identify code block languages** before inserting code examples

### 3.1 AI Agent Guidelines for Markdown Editing

### While Generating Text (Prevention)

- Apply 119-char wrapping to regular prose and list text.
- Do not force-wrap long code, links, or table cells just to satisfy MD013.
- Use abbreviations when they improve readability.
- Prefer line breaks at punctuation or clause boundaries.

## 4. Other Guidance (trimmed for brevity in this synced copy)

Refer to the repository `.instructions.md` and `.prompt.md` for detailed agent workflow and prompt templates.

- Never write `[text](#)` as placeholder.

### While Editing Existing Files

1. Check `.markdownlint.yaml` in the repo root
2. Read surrounding file context before edits
3. Plan line wrapping before touching long prose

### 3.2 When Creating Tables

- Keep column text concise
- Split or abbreviate cells when prose becomes hard to read

### 3.3 When Adding Code Examples

- Always include a code-fence language
- Use `text` if language is unclear
- Ensure fenced blocks are properly closed

### 3.4 When Adding Links

- Verify that file paths and heading anchors exist
- Use relative links for internal references
- Never use placeholder links such as `[text](#)`

### 3.5 Before Committing

### 3.6 Navigation Context and Breadcrumb Decision

**Decision:** Do not use manual breadcrumb lines such as
`LPO > System Architecture > ...` at the top of content pages.

Use this pattern instead:

1. Keep the `# H1` title as the first line
2. Add one short context sentence below the H1
3. Keep navigation in `Child Pages` plus hidden `toctree`
4. Keep a `Table of Contents` block when helpful

**Always run this command:**

```bash
cd /path/to/repo && pre-commit run --all-files
```

**If linting fails:**

1. Read error output carefully (file path, line number, rule code, actual length)
2. Read the specific lines from the file
3. Apply targeted fixes using multi-line context (include 3-5 lines before/after)
4. Run pre-commit again to verify

## 4. Separation of Concerns: Repository Docs vs Content Docs

### 4.1 Repository-Level Documentation

**Purpose:** Manage repository governance, build setup, and contribution process.

**Files:**

- `README.md` — repo overview, strategic direction, links to docs and CI/CD
- `CONTRIBUTING.md` — contribution workflow and style guide
- `CHANGELOG.md` — release history
- `doc-engine.md` — Sphinx build system setup
- `.AI-GUIDELINES.md` — AI/agent editing standards
- `.instructions.md` — agent behavioral constraints

**Key Rule:** Repository docs MAY reference content docs (e.g., README links to project/, requirements/)
but are NOT referenced back from content docs.

**Anti-Pattern:** Do NOT link from `doc-engine/source/project/`, `doc-engine/source/requirements/`,
or other content pages back to `README.md`, `doc-engine.md`, `CONTRIBUTING.md`, etc.

### 4.2 Content Documentation

**Purpose:** Contain the engineering model, requirements, architecture, and validation scope.

**Locations:**

- `doc-engine/source/project/` — Cross-team engineering model, ownership, domains
- `doc-engine/source/requirements/` — Feature and requirement pages
- `doc-engine/source/system-architecture/` — Architecture and design decisions
- `doc-engine/source/system-validation/` — Test evidence and validation scope
- `doc-engine/source/requirement-sources/` — Input artifacts and references

**Key Rule:** Content docs reference other content docs (cross-links within `doc-engine/source/`)
and MAY reference external documentation (LPO baseline, supplier specs, GitHub URLs).

### 4.3 Direction of References

```text
README.md ──points to──> project/project.md
  └──────> doc-engine/source/requirements/
  └──────> doc-engine/source/system-architecture/

project.md <──no backlink──
requirements/ <──no backlink──
system-architecture/ <──no backlink──
```

**Exception:** GitHub-style permalinks (<https://lis-github.liebherr.com/>...) are acceptable
when referencing external repos or repo-level docs as reference material (e.g., linking to
LPO baseline for process guidance).

## 5. Document Structure Standards

All feature and requirement pages should follow this two-section structure:

### Section 1: Main Content (Stakeholder-Facing)

**Contents:**

- Header/title
- Metadata (Initiative, Epic, Category, Context, etc.)
- Overview
- Focus/Scope (phases, boundaries, strategy)
- Organization/Roles
- Acceptance Criteria
- Key Technical Sessions/Milestones

**Guideline:** This section should be self-contained and readable without needing to scroll
past internal process details.

### Section 2: Technical Details (Sphinx/Internal)

**Contents:**

- Implementation planning
- Issues & Lessons Learned
- Controlled Input Baseline
- Traceable Source Synchronization
- References (external links, PDFs, etc.)
- Technical Reference
- Stories/Subtasks toctree
- Processing notes and metadata

**Boundary Marker:**

Use HTML comment to clearly separate:

```html
<!-- ============================================================================
     END OF MAIN FEATURE CONTENT (copy-pastable above this line)
     BEGIN SPHINX / TECHNICAL DOCUMENTATION (internal process & references)
     ============================================================================ -->
```

## 6. Commit Message Guidelines

- Use descriptive commit messages in imperative mood
- Reference which files/sections were modified
- If fixing lint errors, prefix with `docs:` and list the error types fixed

**Example commit:**

```text
docs: update LPO architecture documentation

- Reorganize requirement sections per documentation standard
- Add cross-references between domain areas
- Fix markdownlint violations (MD013, MD040)
```

## 7. Troubleshooting Pre-Commit Failures

### Issue: "files were modified by this hook"

Cause: A pre-commit hook auto-formatted files.

Resolution:

```bash
git diff
git add <files>
pre-commit run --all-files
git commit -m "..."
```

### Issue: "Line length exceeded"

Steps:

1. Check file path and line number from pre-commit output
2. Read the affected line and nearby context
3. Break at punctuation or clause boundaries
4. Re-run pre-commit

### Issue: "Fenced code blocks should have a language specified"

Steps:

1. Locate the unlabeled fence
2. Identify content language (`bash`, `yaml`, `json`, `text`, ...)
3. Add language identifier after opening backticks
