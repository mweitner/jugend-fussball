# Instructions for AI Agents (Copilot, Liebherr AI, etc.)

**Purpose:** Provide context, constraints, and workflows for AI-assisted development in LPO.
**Effective:** May 25, 2026
**Scope:** All markdown editing, documentation generation, and commit assistance.

## 1. Core Principles

- **Autonomous documentation editing** – AI can independently create/edit markdown files
    following `.AI-GUIDELINES.md` rules
- **Human review on code** – All code changes require explicit human approval before commit
- **Pre-commit validation mandatory** – Always run `pre-commit run --all-files` before suggesting commits
- **Prevent errors at generation time** – Avoid creating issues that need fixing; follow linting rules _while writing_

## 2. What AI Agents CAN Do Autonomously

✅ **Documentation & Markdown:**

- Create/edit `.md` files in `doc-engine/source/` and `docs/`
- Fix markdown linting violations (MD013, MD040, MD024, MD036, MD042)
- Reorganize document structure
- Generate tables of contents and navigation
- Generate index and overview helper pages when required
- Add cross-references and internal links
- Suggest commit messages for documentation changes

✅ **Configuration & Templates:**

- Create GitHub issue templates
- Generate documentation templates and examples
- Generate CODEOWNERS templates
- Create Sphinx configuration examples
- Add pre-commit hook configurations

✅ **Content Management:**

- Consolidate duplicate documentation
- Move content between files for better organization
- Update table entries across related documents
- Synchronize related docs (for example roadmap/checklist pairs)

## 3. What AI Agents CANNOT Do (Requires Explicit Request)

🔴 **Code Changes:**

- Create/modify `.py`, `.sh`, `.js`, `.ts` files without explicit instruction
- Modify build scripts or automation unless explicitly requested
- Generate or modify `.github/workflows/` without approval

🔴 **Critical Configuration:**

- Modify `.pre-commit-config.yaml` without explicit approval
- Change `.markdownlint.yaml` rules without justification
- Edit `.github/CODEOWNERS` without team approval
- Modify Python dependencies or `requirements.txt`

🔴 **Security & Compliance:**

- Never generate or suggest credentials, API keys, tokens, or secrets
- Do not modify security configurations
- Do not add external URLs without verifying they are approved for this project

## 4. Markdown Generation Rules (Prevent Linting Errors)

### Rule 4.1: Line Length (MD013 – 119 chars for regular text)

Configured exemptions in this repo:

- Code blocks (`code_blocks: false`)
- Links and URLs (`links: false`)
- Tables (`tables: false`)

- Keep regular prose and list text concise (target <= 119 chars)
- Prefer shorter lines for readability
- For long titles, use abbreviated links
- Do not force-wrap code blocks, URLs, or table rows when it hurts clarity

### Rule 4.2: Code Block Language (MD040)

- **EVERY TIME YOU CREATE A CODE BLOCK:** add language identifier
- Never generate bare `` ``` ``
- Identify content type: `bash`, `yaml`, `python`, `json`, `markdown`, `text`, `rst`, etc.
- If uncertain, use `text`

### Rule 4.3: Emphasis vs. Heading (MD036)

- **NEVER USE:** `*Title*`, `_Title_`, or `**Bold Title Instead of Heading**`
- **ALWAYS USE:** `## Section Heading`, `### Subsection`

### Rule 4.4: Duplicate Headings (MD024)

- **WHEN CREATING SIMILAR SECTIONS:** Add context/qualifier:
  `### What matters most (for Project A)` vs `### What matters most (for Project B)`
- Never repeat exact heading text in same document

**Practical agent rule:** If you prefer to keep visible subheadings short and identical for readability, make the
source unique by appending an invisible HTML comment or an empty span. Examples:

- `#### ✅ Advantages <!-- variant: A -->`
- `#### ✅ Advantages <!-- variant: B -->`
- `#### ✅ Advantages <span id="advantages-a"></span>`

This preserves output while satisfying MD024; use visible qualifiers only when they add reading value.

### Rule 4.5: Link Validation (MD042)

- **EVERY LINK MUST HAVE:** valid target file or heading ID
- Never use `[text](#)` – Always: `[text](#heading-id)` or `[text](./file.md)`
- Verify relative paths: `./file.md`, `../folder/file.md`

### Rule 4.6: Page Context Without Manual Breadcrumbs

Do not add manual breadcrumb lines at the top of content pages.

Use this pattern:

1. `# H1` title
2. one short context sentence below H1
3. hidden `toctree` for navigation
4. optional in-page table of contents when useful

### Rule 4.7: Unnumbered Main Headings

- Do not number main section headings manually.
- Prefer `## Introduction` over `## 1. Introduction`.
- Keep in-page table of contents entries unnumbered to match heading anchors.

## 5. AI Agent Workflow for Documentation Tasks

```text
Task Request
    ↓
[AI Agent Reads Request & Context]
    ↓
[AI Agent Reviews .AI-GUIDELINES.md Rules]
    ↓
[AI Agent Generates Markdown Following Rule 4.x (Prevention)]
    ↓
[AI Agent Runs pre-commit Locally (Simulated Check)]
    ↓
[AI Agent Presents Changes to Human with Commit Suggestion]
    ↓
Human Approves/Modifies ← [HUMAN DECISION GATE]
    ↓
[Human Runs: git add + git commit + pre-commit run]
    ↓
[Human Pushes to Branch]
```

## 6. Workflow for Code Changes

```text
Code Change Request
    ↓
[AI Agent Asks: "Need explicit approval?"]
    ↓
[Human Approves Task Explicitly]
    ↓
[AI Agent Generates Code]
    ↓
[AI Agent Notes: "Code generated; awaiting human review"]
    ↓
Human Reviews Locally ← [HUMAN REVIEW GATE]
    ↓
Human Tests, Validates, Commits
    ↓
Human Pushes
```

## 7. File Categories & Permissions

| Category | File Pattern | AI Can Edit | Requires Approval |
|----------|--------------|-------------|------------------|
| Documentation | `doc-engine/source/**/*.md`, `docs/**/*.md` | ✅ Autonomous | No |
| Markdown Linting | `.markdownlint.yaml` | ❌ No | Yes |
| Pre-Commit Config | `.pre-commit-config.yaml` | ❌ No | Yes |
| GitHub Templates | `.github/ISSUE_TEMPLATE/*.md` | ✅ Autonomous | No |
| CODEOWNERS | `.github/CODEOWNERS` | ⚠️ Ask first | Yes |
| Build Scripts | `Makefile`, `*.sh` | ❌ No | Yes |
| Python Code | `*.py` | ❌ No | Yes |
| Workflows | `.github/workflows/*.yml` | ❌ No | Yes |

## 8. When AI Should Ask for Clarification

- File or directory doesn't exist: "File `/path/to/file.md` not found. Should I create it?"
- Ambiguous instruction: "Unclear whether to abbreviate table text or add column. Suggest: abbreviate. Proceed?"
- Config change: "Change `.markdownlint.yaml` line-length to 100? Requires approval."
- Cross-team impact: "This change affects CODEOWNERS. Needs review from approval team?"

## 9. Commit Message Guidelines for AI-Suggested Changes

**Format:**

```text
<type>: <short description> [AI-ASSISTED]

- Point 1 (what changed)
- Point 2 (why)
- Point 3 (related files)
```

**Examples:**

✅ Documentation:

```text
docs: reorganize LPO architecture sections [AI-ASSISTED]

- Organized requirement domains by system components
- Added cross-system integration references
- Updated navigation and toctrees
```

✅ Content consolidation:

```text
docs: consolidate duplicate LPO documentation [AI-ASSISTED]

- Merged redundant architecture definitions
- Updated links to single source of truth
- Files: requirements.md, architecture.md
```

## 10. Error Recovery

**If AI-generated markdown fails pre-commit:**

1. AI provides the exact error output
2. Reference `.AI-GUIDELINES.md` section 2 (linting rules)
3. AI suggests specific fix with line numbers and context
4. Human approves fix, runs pre-commit locally
5. Human commits

Expected: Should rarely happen if Rule 4 (prevention) is followed.

- **Documentation structure:** See provided standards for content organization
- **Pre-commit hooks:** `.pre-commit-config.yaml` defines all automated checks
